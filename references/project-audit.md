# Project Audit

Use this file before concept generation whenever the project already exists.

The goal is to prevent disconnected logo or app icon directions that ignore the product already in front of you.

## Audit Order

Inspect local project artifacts first:
- product docs
- landing pages or in-app copy
- screenshots
- existing logo and icon files
- `.pen` boards
- Figma frames
- design tokens or brand variables
- current UI components
- app store assets if available

## Build A Style Snapshot

Summarize the current product in a compact style snapshot:
- product promise
- emotional tone
- category cues already in use
- dominant shapes
- current color logic
- typography voice
- level of polish: utilitarian, warm, premium, playful, technical, etc.
- what already feels strong
- what currently feels weak, generic, or inconsistent

## Redesign Tolerance

Classify the redesign scope:
- evolutionary
  Keep recognizable equity and refine the existing system.
- adjacent
  Preserve the product's tone, but improve distinctiveness and craft.
- reset
  Only use when the user explicitly wants a break from the current direction, or the current direction is clearly failing.

Default assumption:
- if the project already has a coherent style, start with evolutionary or adjacent directions first

## Alignment Rules

When the current product has a clear design language:
- reuse its emotional tone unless the user asks to change it
- keep the icon's geometry compatible with the UI's geometry
- do not invent a premium/material language if the product looks strictly flat and utilitarian
- do not force soft/playful symbolism onto a severe product
- do not force severe symbolism onto a warm, friendly product

Useful signals:
- corner radius
- stroke weight
- fill vs outline preference
- contrast level
- density and detail tolerance
- symmetry vs asymmetry

## Mandatory Outputs From The Audit

Before ideation, be able to state:
- what must be preserved
- what can change
- what the current system is missing
- what a successful next-step logo/icon direction must solve

If the audit surface is thin:
- say that the project context is weak
- switch to category and platform-driven exploration
- keep assumptions labeled
